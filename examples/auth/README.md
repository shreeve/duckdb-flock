# harbor auth callback recipes

The default `harbor_authentication_function` accepts the static token
generated by `harbor_serve()`. That's fine for development — but
production deployments almost always need something stricter:
multi-tenant token tables, RBAC, time-limited tokens, etc.

This directory has copy-paste-ready SQL examples for the most common
patterns. Each file is self-contained and can be loaded into a running
harbor with `.read examples/auth/<file>.sql`.

> **Important:** auth callbacks must be set with `SET GLOBAL`, not
> plain `SET`. The auth path runs in fresh transient connections that
> don't see session-local settings. A plain `SET` silently has no
> effect on authentication.

## Recipes

| File | What it gives you |
|---|---|
| [`bearer-only-static.sql`](bearer-only-static.sql) | Single shared bearer token loaded from a settings table. Simplest production case. |
| [`bearer-table-multi-tenant.sql`](bearer-table-multi-tenant.sql) | Token-per-tenant table with active/revoked state. Each token maps to a distinct principal. |
| [`bearer-with-expiry.sql`](bearer-with-expiry.sql) | Tokens carry a `valid_until` timestamp; expired tokens are rejected without manual revocation. |
| [`rbac-authorization.sql`](rbac-authorization.sql) | Authorization function that checks a `principals_admin_actions` table for `__HARBOR_ADMIN__:resource:action` queries — locks down admin endpoints. Pairs with any of the bearer recipes above. |

## Why these are SQL files

`harbor_authentication_function` and `harbor_authorization_function`
take the **name** of a SQL function, not function bodies. So every
auth recipe is a `CREATE OR REPLACE FUNCTION ... AS BEGIN ... END`
statement that lives in your database catalog and gets called by harbor
on every credential check / authz check.

They run in transient connections (no session state), so they must be
deterministic on their inputs and pure with respect to whatever tables
they consult.

## Operator runbook

After loading any of these recipes:

```sql
-- Verify it's actually wired in:
SELECT current_setting('harbor_authentication_function');
SELECT current_setting('harbor_authorization_function');

-- The (DESCRIBE FUNCTION) trick to confirm signatures are what harbor expects:
SELECT * FROM duckdb_functions() WHERE function_name LIKE 'my_%_auth%';

-- Restart harbor for the GLOBAL settings to take effect on running:
CALL harbor_stop();
CALL harbor_serve('harbor:127.0.0.1:9494');
```

After re-`harbor_serve`, the new auth functions take effect immediately.
The previously-issued `harbor_session` cookies still work (cookie HMAC
is signed against the ephemeral per-process key, not the auth function),
but new `/auth/login` calls and every Bearer-credentialed request will
go through your new callbacks.
